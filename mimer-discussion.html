Mimer Project Discussion

 

Introduction

This was a great project. One of those we-will-laugh-one-day-about-this projects that you don’t really believe it until you’re finally laughing about it. It tested our proficiency of concepts like marshaling, differences and relationships between a MIME-type and an extension, servers with back channels to receive data through other ports, and writing to disk files. But it also tested our patience, perseverance, and ability to ask relevant questions. At the very least tested our courage to ask questions to our peers.

 

The process play by play

These are the steps I took to complete the assignment:

 

1.     Updated MyWebServer to send “Content-Type: application/xyz” header to the browser. Tested the updated code.

2.     Translated shim.bat to shim.sh: this took quite some time because I was starting from the absolute bottom. I didn’t know what batch files were. Found out they are just shell scripts with instructions so you don’t have to type everything into the terminal line by line. Then, had to find out if there was an equivalent in OSX, found out about bash scripts. The tips started to some sense at that point. And finally, had to figure out the syntax to accomplish the same as the .bat file for a .sh file.

3.     Associating the MIME-type with the handler: this step took an unreasonable amount of time. I was not familiar with how OSX relates MIME-Types to a handler. I went down so many different roads including articles that described how to create a dictionary for an extension using Apple Launch Services, writing a script with Apple’s Automator program, or writing a script with Apple Scripts. Each one with hours of background reading and trail-and-error testing. With help from my classmates I was pointed in the right direction to write the script using Apple Scripts. I had to get familiar with the syntax to adjust and debug the snippet from a prior student in the tips section. Lot’s of learning in this tiny step.

4.     Setting environment variables: this helped me run the shell scripts without having to type the whole path.

5.     Added and compiled the Handler.java.

6.     Tested calling “shim mimer-data.xyz” which printed to the console but data was not printing into the temp file at all. New files were being created but with no data.

7.     Tested some more: mimer-data.xyz using my browser through mimer-call.html. I finally figured I wasn’t using the right class in MyWebServer to print to a temporary file. I was using DataOutputStream to write bytes to the browser, but this class is not able to print to a file. I eventually added a PrintStream class to accomplish this. New terminal window running my script and calling the handler started running with every call to mimer-data.xyz from the browser.

8.     Wrote many comments on Handler and BCClient, all of which were later used to comment the stitching together of BCHandler.

9.     Downloaded appropriate jars

10.  Created scripts to compile and run the various classes. Every script has a path to find the downloaded jars, therefore the java files do no compile without the script.  

11.  Played around with the scripts: I had some bugs like wrong paths to where my classes were, wrong jar names, and accidentally wrote the java command more than once which prints the documentation for that command on the terminal. I also got “permission denied message” which I fixed with the command “chmod +x pathname”

12.  Added back channel administrator with its BCWorker thread class.

13.  Combined BCClient and Handler into combo class BCHandler: Just when I thought I was getting to a smooth sail to the end I had to figure one last challenge. I think the stitching together of the two classes is what best helped me understand what the back channel does connection does. It is really similar to our mode administrator in the joke server assignment, but in this case, we are emphasizing the fact that the back channel can receive data and send acknowledgement of this receipt.  

14.  Test and debug: This was not too bad after the incredible time investment of the prior tests. Downloaded to folders from D2L and made sure they are not corrupted.

 

 

Comments

I am very glad we get to do a write-up on this project because it was overall a rollercoaster of emotions. Partly, I was for hoping something easier since this assignment was competing for time with the midterm. Also the instructions say things can take 4 minutes or 4 hours and at every step I kept hoping it was going to take 4 minutes. Although eventually I just hoped for at most 4 hours. Now that I’m done I’m glad I invested the time researching and thinking about each part of the assignment and how it all comes together.

 

I took the advice of “not expecting everything to work right away” to heart. In many areas I wasn’t sure if the program was behaving correctly since so much was new to me. I had to take leaps of faith and keep trucking along. In time I did realize there were some things I was doing wrong, but I was only able to recognize them because I kept going and kept learning more about the how the code worked.

 

I also liked that for this assignment folks were happy to help one another. I felt lost really early in the process and I know for a fact I could not have made it without the support of my peers.

 

This assignment was hard but I wholeheartedly agree that it’s a worthwhile exercise with much to be gained.
